# Metrics catalog

> Generated by `backend/scripts/generate_metrics_catalog.py` — re-run after
> registry changes. Do not edit by hand.

Every metric surfaced to the UI registers a `MetricSpec` in
`backend/app/services/metric_spec.py`. The fields below drive:

- Tooltip and paired-outcome wiring on stat cards
- Visibility defaults (`self` / `team` / `admin`)
- Distribution-style rendering for metrics requiring p50 + p90 (histogram)
- Goodhart risk disclosure so readers know what each number distorts when
  optimized for in isolation

Activity metrics (`is_activity=True`) must declare a `paired_outcome_key`;
this is enforced at import time by `MetricSpec.__post_init__`.

## Banned metrics

Metrics DevPulse explicitly does not expose. Recorded here so the decision is visible and reversible.

| key | reason |
|-----|--------|
| `lines_of_code_per_dev` | LOC is not a productivity metric. Correlates negatively with quality. |
| `commits_per_dev` | Incentivizes commit-padding and discourages meaningful commit hygiene. |
| `story_points_per_sprint_per_dev` | Per-individual velocity is weaponizable and blind to scope, complexity, blockers. |
| `time_to_first_review_as_kpi` | Creates incentives for rubber-stamp reviews. Ship distribution, not target. |
| `loc_weighted_impact_score` | Amplifies LOC bias. Reviewer-count or bus-factor is a better impact proxy. |
| `raw_sentiment_per_developer` | Cross-cultural noise ~30% per research. Team-aggregate only, opt-in. |
