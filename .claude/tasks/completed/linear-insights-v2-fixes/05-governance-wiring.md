## Description

Phase 11 of the original epic shipped three governance components as dead code —
`DistributionStatCard`, `AiCohortBadge`, and the `StatCard.pairedOutcome` slot all exist but
no page imports or uses them. The `MetricsUsageBanner` is wired to Insights routes only,
missing Dashboard and Executive. And `docs/metrics/catalog.md` (an explicit Phase 11
deliverable) was never created.

This phase activates the governance work so Phase 11's intent actually lands in the UI.

## Deliverables

### `frontend/src/App.tsx` — `MetricsUsageBanner` on Dashboard + Executive routes

**Bug** (line 174 vs lines around Dashboard + Executive routes): the banner is wrapped around
the `/insights/*` SidebarLayout only. Dashboard (`/`), Executive (`/executive`), and admin
metric pages (`/admin/linkage-quality`, `/admin/metrics-governance`,
`/admin/classifier-rules`) do not render it.

**Fix**: render `<MetricsUsageBanner className="mb-4" />` at the top of:
- `pages/Dashboard.tsx` (inside the main content wrapper)
- `pages/ExecutiveDashboard.tsx`
- `pages/admin/LinkageQuality.tsx`
- `pages/admin/MetricsGovernance.tsx` — already renders it inline on line 67; verify the
  Admin layout root does NOT wrap with the banner (it doesn't) so the inline usage is correct
- `pages/admin/ClassifierRules.tsx`

The banner's quarterly-dismiss logic in `MetricsUsageBanner.tsx` uses a shared localStorage
key, so multiple render sites share dismissal state — no UX downside to placing on multiple
routes.

### Wire `DistributionStatCard` into DORA v2 + Flow Analytics

**Gap**: `DistributionStatCard` (p50 + p90 + histogram sparkline) has zero import sites. The
three most natural homes:

1. `frontend/src/pages/insights/DoraMetrics.tsx` — replace the Lead Time and MTTR stat cards
   with `DistributionStatCard` when the backend returns p50/p90 (it already does via DORA
   v2). The component expects a histogram; `get_cycle_time_histogram` (from Phase 07) returns
   buckets compatible with the sparkline shape — verify shape match and adapt if needed.
2. `frontend/src/pages/insights/FlowAnalytics.tsx` — the status-time distribution response
   has p50/p90/histogram per state category. Render each state as a `DistributionStatCard`.
3. `frontend/src/pages/insights/Bottlenecks.tsx` — cycle-time histogram card is a direct
   match.

For each wiring, if the backend response shape doesn't match the component's expected props,
either (a) adapt the component to accept the response shape, or (b) massage on the client.
Prefer (a) for cleanliness.

### Wire `AiCohortBadge` into DORA v2 cohort cards + AI analysis pages

**Gap**: zero import sites. The natural home is the cohort comparison card on
`DoraMetrics.tsx` — each of the four cohort rows (Human / AI-reviewed / AI-authored / Hybrid)
should carry an `AiCohortBadge` with the cohort type as the label. Verify the component's
props and adapt if the shape doesn't match.

### Wire `StatCard.pairedOutcome` slot into activity-metric cards

**Gap**: the `pairedOutcome` slot was added to `StatCard` but no existing StatCard usage
passes the prop. Per the MetricSpec principle "activity metrics pair with outcome", any
StatCard displaying an activity metric (PR count, review count, comment count) should pass a
matching outcome.

**Scope (bounded — don't try to wire every StatCard)**: wire the slot on:
- Dashboard PR count card — pair with merge rate or cycle time
- Dashboard review count card — pair with time-to-first-review or review-load-Gini
- DeveloperDetail workload section — pair PR count with merge rate

For each wiring, look up the `paired_outcome_key` declared on the activity metric's
`MetricSpec`, fetch the corresponding outcome value from the same API response or a
sibling endpoint, pass as `pairedOutcome={{ label, value, tooltip }}`.

### `docs/metrics/catalog.md` — generate from the registry

**Gap**: Phase 11 deliverable, never created.

**Fix**: add a small backend script `backend/scripts/generate_metrics_catalog.py` that imports
the `MetricSpec` registry and renders a Markdown table (one row per metric) to
`docs/metrics/catalog.md`. Columns: key, category, is_activity, paired_outcome_key,
visibility_default, is_distribution, goodhart_risk, goodhart_notes, description.

After the registry, render a second table for `BANNED_METRICS` with columns: name, reason.

Commit the generated file. Add a note at the top of the file: "Generated by
`scripts/generate_metrics_catalog.py` — re-run after registry changes."

Optional (consider): pytest that asserts the committed `catalog.md` matches the freshly-
regenerated output, so drift is caught in CI.

## Implementation notes

- `DistributionStatCard` usage on DoraMetrics was deferred: DORA v2 returns averages,
  not p50/p90. Wired on FlowAnalytics (`status-time distribution` already returns
  `p50_s`/`p90_s` per state) and Bottlenecks (`cycle-time histogram` returns p50/p90).
- AiCohortBadge replaces the hand-built `<Badge>` in the AI-touched disclosure strip
  on DoraMetrics — same hover disclosure, same drill target, now from the shared
  component.
- `generate_metrics_catalog.py` imports the `REGISTRY` and `BANNED_METRICS` lists and
  emits a markdown table per row. Pattern matches the existing `backend/scripts/*.py`
  layout. No CI-drift check added; the task spec flagged this as optional and I left it
  for a follow-up with a proper pytest.
